Project 01

Kafka 기반 분산 시스템 운영·관측·부하 검증

2025.05 – 2025.06  ·  개인 프로젝트  ·  GitHub

지표 기반의 워크로드 분석으로 컨슈머·파티션 배치 불균형을 발견하고, 분산 배치를 통해 특정 인스턴스의 CPU 부하를 10% 완화했습니다. 브로커 장애 실험을 통해 메시지 손실 없는 서비스 지속 가능성을 직접 검증했습니다.

배경 및 목표

단순히 Kafka를 설치하고 메시지를 주고받는 수준을 넘어서, 실제 운영 환경처럼 분산 구성을 갖추고 그 위에서 시스템이 어떻게 동작하는지를 직접 관찰하고 싶었습니다.

특히 관측 가능성의 개념을 직접 경험해보는 것이 핵심 목표였습니다. 부하를 가하고, 지표를 보고, 문제를 찾고, 개선하는 사이클을 한 번 완성하는 것을 목표로 설정했습니다.

아키텍처 선택 — 트레이드오프

멀티 브로커 구성을 위해 AWS EC2(t3.medium) 3대를 선택했습니다. 브로커 간 통신, 리더 재선출, ISR 변화를 더 실제 운영 환경에 가깝게 관찰하기 위해 서로 다른 EC2 인스턴스에 브로커를 분산 배치했습니다.

모니터링 스택은 처음에 Prometheus로 구성했다가 VictoriaMetrics로 전환했습니다. 초기에는 Prometheus로 지표를 수집했지만, 이 프로젝트에서는 장기 저장보다 경량 수집과 원격 저장 구조가 더 중요하다고 판단했기에 수집은 vmagent, 저장은 VictoriaMetrics로 분리했습니다. vmagent가 Prometheus 스크래핑 설정을 그대로 사용할 수 있어 기존 exporter 생태계를 그대로 유지하면서 모니터링 구조를 단순화할 수 있다는 점도 함께 고려했습니다.

구현 내용

  • EC2 3대에 Docker Compose로 Kafka 멀티 브로커 클러스터 구축. 각 노드에 kafka-exporter, cAdvisor 배포
  • vmagent가 각 노드의 지표를 수집해 VictoriaMetrics로 전달, Grafana에서 Lag·Latency·컨테이너 리소스를 단일 대시보드로 통합
  • 쿠폰 발급 요청을 받은 Django API 서버가 Kafka에 메시지를 전송하고, 컨슈머 컨테이너 6개가 메시지 처리 구조 구성. 개선을 거쳐 최종적으로 EC2에 컨슈머 2개씩 분산 배치.
  • k6로 5,000 / 10,000 / 15,000 / 18,000건 단계별 부하 테스트 진행. 싱글 브로커 → 멀티 브로커 전환 전후를 동일 조건(180초, 15,000건)에서 비교해 수치로 검증

문제 발견 → 분석 → 개선

1. 워크로드 편중 발견
멀티 브로커로 전환한 뒤 동일 조건으로 부하 테스트를 재실행했는데, 인프라를 확장했음에도 p95 응답 지연이 오히려 약 9.4배 증가했습니다. 자원과 메시지 처리 단위를 늘렸는데도 성능이 떨어졌다는 점이 이상해 Grafana 대시보드를 점검했고, EC2 인스턴스 간 CPU 사용률 편차가 크다는 것을 발견했습니다. 특정 인스턴스가 78%까지 치솟는 동안 다른 인스턴스는 20% 수준에 머물렀습니다. 컨슈머 배치 현황을 확인하자 원인이 명확해졌습니다. 총 6개의 컨슈머 컨테이너가 모두 같은 EC2 인스턴스에 몰려 있었고, 파티션의 리더도 같은 브로커에 집중된 상태였습니다.

2. 분산 배치로 개선
각 EC2에 브로커 1개 + 컨슈머 2개씩 균등 배치하고, 파티션 리더도 분산되도록 조정했습니다. 이후 동일 조건으로 부하 테스트를 재실행하자 응답 지연(p95)이 2.06s에서 33.3ms로 단축됐고, 피크 CPU 사용률이 78% → 68%로 낮아졌습니다. 가장 부하가 높았던 컨테이너 기준으로는 약 47% 완화되었습니다.

3. 추가 병목 식별
분산 배치 이후에도 최대 부하 구간에서 응답 지연이 남아 있었습니다. 쿼리 실행 로그를 분석한 결과 MySQL I/O 비용이 병목임을 확인했습니다. 메시지 처리마다 DB 쓰기가 발생하는 구조였기 때문에, 컨슈머 처리량이 늘어날수록 DB가 따라오지 못하는 상황이었습니다. 서비스 확장 시, DB 전용 EC2 분리를 개선 방안으로 정리했습니다.

장애 내성 검증

부하 테스트 중 브로커 1대를 강제 종료해 장애 상황을 재현했습니다. 리더 재선출 과정과 ISR(In-Sync Replica) 변화를 실시간으로 관찰했고, 메시지 손실 없이 서비스가 지속됨을 확인했습니다. 테스트 전후 producer 발행 건수와 consumer 처리 완료 건수, DB 저장 건수를 비교해 누락 여부를 확인했습니다. 이 과정에서 acks=all, replication.factor=3, min.insync.replicas=2 설정이 실제로 어떻게 동작하는지 체감할 수 있었습니다.

구현 확장 - Kubernetes 전환 및 CI/CD 구성

  • Docker Compose 기반 구성을 Kubernetes로 전환. Kafka 클러스터는 Strimzi Operator의 Custom Resource로 선언해 StatefulSet·Service·PVC 생성 자동화.
  • 애플리케이션 코드는 수정하지 않고 서비스 간 통신 주소를 EC2 IP에서 Kubernetes Service DNS로 변경하고, 환경변수를 ConfigMap으로 분리.
  • GitHub Actions로 main 브랜치 push 시 Docker 이미지를 빌드해 GHCR에 push. 커밋 해시 기반 이미지 태그를 Kubernetes manifest에 반영하도록 구성.
  • kubectl apply와 rollout status 확인을 자동화해 코드 변경부터 컨테이너 이미지 갱신, Kubernetes 배포 반영까지 이어지는 배포 흐름 재현.

Project 02

Serverless VPC 네트워크 설계와 비용 구조 비교 분석

2025.03  ·  개인 프로젝트  ·  GitHub

서버리스 자원을 활용한 비동기 파이프라인을 설계하고, NAT Gateway와 VPC Endpoint 두 아키텍처의 비용 구조를 직접 비교 분석했습니다. AWS CDK로 IaC를 구현하고, VPC Endpoint 전환 시 최소 36% 비용 절감 시나리오를 도출했습니다.

배경 및 목표

이벤트 예약 시스템은 특정 시간대에 트래픽이 집중되는 특성이 있습니다. 서버를 상시 운영하면 대기 비용이 발생하고, 트래픽 급증 시 가용성을 보장하기 어렵습니다. 이 문제를 해결하기 위해 트래픽에 따라 자동 확장되는 서버리스 자원을 선택했습니다. 설계 과정에서 Private Subnet에 배치한 Lambda가 AWS 관리형 서비스에 접근하는 경로로 NAT Gateway와 VPC Endpoint 두 가지를 검토했고, 두 방법의 비용 구조 차이를 실제 수치로 비교해보고 싶었습니다.

아키텍처 설계 — 트레이드오프

Lambda는 Private Subnet 배치를 선택했습니다. 이 경우 Lambda가 SQS·SNS·DynamoDB 같은 AWS 관리형 서비스에 접근하기 위한 네트워크 경로가 필요합니다. NAT Gateway를 통한 아웃바운드 경로와 VPC Endpoint를 통한 프라이빗 연결 경로를 비교했습니다.

Option A — NAT Gateway

  • 구성이 단순하고 범용적
  • 인터넷을 경유해 AWS 서비스 접근
  • 시간당 고정 비용 + 데이터 처리 비용 발생
  • 트래픽이 늘수록 증가 폭이 커짐

Option B — VPC Endpoint ✓ 선택

  • 서비스별·AZ별 Endpoint 구성 필요
  • AWS 내부 망 경유, 인터넷 미경유
  • AZ당 고정 비용 + 데이터 처리 비용 발생
  • 트래픽이 많을수록 절감 폭이 커짐

VPC Endpoint는 구성 복잡도가 높지만, 해당 조건에서 약 36%의 비용 절감 가능하며 낮은 데이터 처리 비용 덕에 트래픽이 클 수록 절감 폭이 커집니다.

구현 — 파이프라인 및 안정성

  • API Gateway → Lambda(예약 요청) → SQS FIFO → Lambda(예약 처리) → DynamoDB 비동기 파이프라인. Lambda 처리 실패 시 SQS 자동 재전달로 안정성 확보
  • SQS FIFO에서 좌석 기준으로 메시지 그룹을 분리해 동일 좌석의 요청을 순차 처리
  • Request ID + DynamoDB 조건부 업데이트로 재시도·중복 요청에도 멱등성 보장
  • SQS Batch Size 1→5 조절 및 앞단 Lambda에 Provisioned Concurrency 적용으로 Cold Start로 인한 초기 응답 지연 완화
  • AWS CDK로 전체 인프라를 NetworkStack, ApplicationStack, DatabaseStack 등 6개 스택으로 분리해 IaC 구현
  • CloudWatch Alarm과 SNS 이메일 구독으로 운영 알람 구성
